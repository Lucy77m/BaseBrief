# BaseBrief Known Limitations

BaseBrief is a small local handoff toolchain. These limitations are intentional unless real usage demonstrates a need to change them.

## Input

- The handoff builder accepts BB9 JSON or Markdown containing the marked structured JSON block.
- Free-form Markdown is not automatically converted into structured handoff data.
- Facts, decisions, risks, and open questions still require human or agent review.

## CLI

- CLI Lite requires explicit input and output paths.
- It does not scan a project automatically, maintain background state, or modify host-tool configuration.
- It is a repository script, not an installed global command or published package.

## Checks

- The artifact checker detects a small set of deterministic publication hazards.
- It cannot prove that an artifact contains no secret, private detail, incorrect fact, or misleading claim.
- Warnings require review even though they keep a zero exit code.

## Seal/Diff

- Seal/Diff compares structured BaseBrief handoff fields.
- It is not a Git diff, source-code audit, project timeline, or release system.
- It does not detect repository changes that are absent from the handoff data.

## Provider Evidence

- Provider-facing behavior follows explicit provider profiles and fallback rules.
- Estimated-cost evidence remains scoped to the tested provider, model, fixtures, and benchmark conditions.
- BaseBrief does not perform billing audits or guarantee cost, cache, or latency outcomes.

## Integrations

- Codex and Claude adapters are file exports, not official integrations or plugins.
- BaseBrief does not automatically send artifacts to an AI tool or provider.

## Receiver Environment

- BaseBrief can carry target-repository context, receiver entry tasks, and risk boundaries, but it does not automatically switch the receiver's working directory.
- Cross-directory startup still requires the user or host tool to provide a private path to the handoff artifact.
- Explicit receiver-ready fields reduce ambiguity, but BaseBrief cannot eliminate differences in model reasoning capability or guarantee that every receiver will interpret stale state correctly.

## Receiver Language

- `response_language: match_latest_user_message` controls language generated by the receiving agent, including its first sentence, progress updates, and final report.
- Platform-generated tool traces such as `Viewed` or `Ran command` may remain in the host tool's language.
- Code, commands, paths, and protocol field names remain unchanged even when the surrounding explanation is translated.

## Receiver Acceptance

- `expected_changed_files` enables exact changed-file comparison, but it does not prove that file contents or runtime behavior are correct.
- `handoff_acceptance: difference_found` means the receiver completed its task and accurately reported a state difference; it is not the same as an execution failure.
- Receiver state verification remains distinct from source-window behavioral verification such as tests or release checks.
- Optional Receiver Safe Check can run only the declared `node_syntax`, `artifact_check`, and `file_tokens` checks; it is not a general test runner or proof of behavioral correctness.
- Receiver Safe Check does not execute raw commands, permit declared checks to inspect `.env` or `.git` paths, validate file-content hashes, or automatically decide whether a handoff is stale.
- `generated_at` provides a time reference, but BaseBrief does not automatically decide when a handoff is stale or block acceptance based on age.
- Receiver-ready v1 does not add file-content hashes, merge the next-chat prompt with the readable brief, or change repository line-ending policy.
- Existing LF-to-CRLF warnings may remain as platform noise; they require separate repository-wide evaluation rather than a receiver-protocol workaround.
