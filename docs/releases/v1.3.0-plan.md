# v1.3.0 Delta Receiver Starter Integration Plan

Date: 2026-06-07

This document records the recommended v1.3.0 planning direction after the v1.2
Delta Receiver Report Kit line became locally closeout-ready.

It is a planning baseline, not a push, tag, release, package publication,
provider integration, runtime integration, plugin, MCP, IDE, hosted service,
Auto Flow behavior, global CLI install, npm publication, command output format
change, JSON schema claim, or schema-v2 claim.

## Summary

BaseBrief v1.3.0 should keep the Delta Receiver line local-first and
review-based by integrating the v1.2 report kit into starter-facing docs and
examples:

```text
Delta Receiver Starter Integration
```

The goal is not to create a new runtime or command. The goal is to make the
existing receiver starter path point receivers toward the v1.2 report shape
before they continue implementation.

## Recommended Direction

Prefer a docs/examples/tests slice:

- connect the v1.2 report kit from quickstart, golden-path, and receiver
  examples where a receiver first learns what to report
- keep protocol literals, paths, and acceptance words stable
- preserve source-window inherited facts versus receiver-window rechecks
- keep `difference_found` as a completed receiver outcome, not an agent failure
- keep historical `commits_in_range` drift explainable and non-blocking when
  refreshed branch, HEAD, and worktree facts match

## v1.3.0 In Scope

- Starter-facing documentation polish for the v1.2 report kit.
- Public-safe examples that show where to copy the report shape.
- Release-check and focused test coverage for discoverability and boundaries.

## v1.3.0 Out Of Scope

- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No `basebrief-project-state-v2`.
- No `basebrief-sidecar-v2`.
- No new CLI command.
- No new npm script.
- No machine-readable JSON schema for receiver reports.
- No command output format change.
- No Auto Flow.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output in public docs.

## Success Criteria

v1.3.0 is successful if all of the following are true:

1. A receiver following starter-facing docs can find the v1.2 report kit before
   continuing implementation.
2. Existing receiver examples still distinguish inherited facts from live
   receiver-window rechecks.
3. No schema, command, runtime, provider, plugin, MCP, IDE, Auto Flow, hosted,
   or publication scope is introduced.
4. `provider_probe_status=skipped` remains the no-provider validation gate when
   provider environment variables are absent.

## Deferred Alternatives

Thin command exploration, external manual receiver smoke, provider-backed tests,
and runtime integration remain deferred until repeated local usage shows a
concrete blocker.

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```
