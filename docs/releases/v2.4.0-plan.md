# v2.4.0 File-only Adapter / MCP-friendly Export Plan

Date: 2026-06-08

Status: v2.4-A contract freeze

This document freezes the docs-first v2.4 direction. It does not implement a
command, generator, exporter, JSON schema file, schema-v2, provider request,
runtime integration, plugin, MCP server, IDE integration, hosted service,
cloud-memory behavior, or Workflow Runner.

## Goal

BaseBrief v2.4 defines a future local-only export layer for Context Pack Lite
and BaseBrief Format artifacts.

The planned export family is:

```text
exports/
exports/manifest.json
exports/context-pack.md
exports/context.json
exports/adapter-notes.md
```

The wording should stay restrained: v2.4 is file-only adapter output that
other local tools may consume. It is not an MCP server, plugin, runtime
integration, hosted connector, or universal protocol.

## Contract Direction

The v2.4-A contract is:

- exports are written only to an explicit local output directory
- exports start from checked Context Pack Lite inputs and future reviewed
  BaseBrief Format artifacts
- `exports/context-pack.md` is the readable single-file handoff export
- `exports/context.json` is the machine-readable summary export
- `exports/manifest.json` records source files, export kind, contract version,
  generator identity, and review status
- `exports/adapter-notes.md` records human review notes and tool-consumption
  boundaries
- source, trust, review status, stale, `not_available`, `not_applicable`, and
  `needs-review` semantics must be preserved
- receivers and consuming tools must still recheck live repo facts before
  implementation

## MCP-friendly Meaning

MCP-friendly means file shapes are easy for a future local MCP server or other
tool to read. It does not mean v2.4 creates an MCP server, registers tools,
opens a transport, manages sessions, calls models, or writes host-tool config.

Acceptable v2.4-A language:

- file-only export
- local artifact bundle
- MCP-friendly files
- future tool-consumable shape

Forbidden v2.4-A language:

- MCP server
- plugin integration
- runtime integration
- hosted connector
- cloud memory
- automatic tool execution

## Non-goals

- No command.
- No implementation.
- No generator.
- No exporter.
- No JSON schema file.
- No schema-v2.
- No format emission.
- No provider request.
- No runtime integration.
- No plugin, MCP server, IDE, hosted service, or cloud-memory behavior.
- No Workflow Runner.
- No change to `basebrief-project-state-v1`.
- No change to `basebrief-sidecar-v1`.
- No change to Context Pack Lite generator output.
- No change to `check --input <dir> --json` top-level shape.
- No change to existing `resume --input <context-pack-dir>` behavior.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output, private absolute paths, `.env`, secrets, tokens,
  credentials, API keys, or bearer strings.

## Future Implementation Notes

A later v2.4-B implementation may add a local file-only export command only
after this contract is reviewed.

That later implementation should start from checked Context Pack Lite
directories. It should not invent missing facts, silently refresh inherited
facts, collapse warning/error semantics, call providers, create an MCP server,
or modify host-tool configuration.

The first implementation slice should prefer the existing local CLI wrapper
rather than a new package, daemon, plugin, or hosted surface.

## Validation Gate

The local validation gate for this planning line is:

```text
npm test
npm run release-check
git diff --check
```

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```
