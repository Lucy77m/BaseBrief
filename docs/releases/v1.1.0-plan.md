# v1.1.0 Delta Receiver Acceptance Plan

Date: 2026-06-07

This document records the adopted v1.1.0 planning baseline after the v1.0
Delta Handoff line became locally reviewable.

It is a planning baseline, not a push, tag, release, package publication,
provider integration, runtime integration, plugin, MCP, IDE, hosted service,
or schema-v2 claim.

## Summary

BaseBrief v1.1.0 should extend the Delta Handoff line in the narrowest useful
direction:

```text
Delta Receiver Acceptance Kit
```

The goal is not to create an agent runtime. The goal is to make it easier for
a fresh receiver window to prove that it understood `delta-handoff.md` before
continuing implementation work.

## Receiver Acceptance Minimum

A receiver acceptance response should explicitly restate:

- current goal
- live repository state versus inherited handoff facts
- changed files and worktree state
- hard boundaries
- next narrow implementation slice

The receiver should also report:

```text
receiver_task_status
repository_state_status
handoff_acceptance
blocking_or_repair_notes
```

## v1.1.0 In Scope

- A public-safe planning baseline for receiver acceptance.
- A public-safe dogfooding record that shows how the acceptance response should
  be judged.
- Documentation and release-check coverage that keep the receiver acceptance
  line discoverable.
- Tests that guard the v1.1.0 documents and boundaries.

## v1.1.0 Out Of Scope

- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No `basebrief-project-state-v2`.
- No `basebrief-sidecar-v2`.
- No new CLI command.
- No new npm script.
- No Auto Flow.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output in public docs.

## Success Criteria

v1.1.0 is successful if all of the following are true:

1. A receiver can read a delta handoff and state whether the live repository
   matches the inherited handoff facts.
2. The response separates source-window inherited facts from receiver-window
   rechecks.
3. The response identifies hard boundaries before proposing the next slice.
4. The public dogfooding record remains public-safe and does not copy raw
   private output.
5. `basebrief-project-state-v1`, `basebrief-delta-handoff-v1`, and
   `basebrief-delta-baseline-v1` remain unchanged.

## Current Boundaries

- `provider_probe_status=skipped` remains the correct no-provider gate.
- Generated `.basebrief/out/` artifacts remain local-first and ignored by
  default.
- Delta Receiver Acceptance is a review contract, not an automated runtime.
- Later provider, plugin, MCP, IDE, runtime, hosted, or schema-v2 work must
  remain frozen unless a separate measured blocker justifies it.
