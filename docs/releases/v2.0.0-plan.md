# v2.0.0 Context Pack Lite Plan

Date: 2026-06-08

This is the v2.0.0 planning baseline for BaseBrief Context Pack Lite. It is not
a release, tag, push, pull request, npm publication, global CLI install,
provider request, runtime integration, plugin, MCP server, IDE integration,
hosted service, schema-v2 change, checker rule family, command implementation,
or Workflow Runner.

## Summary

BaseBrief v2.0 starts a new public line:

```text
Context Pack Lite
```

The goal is to define a local, reviewable, file-based context pack for coding
agent handoff. The pack should help a new window understand project identity,
repository shape, key files, recent delta, risk boundaries, receiver state, and
the next safe starter prompt without dumping the whole repository or inventing
missing context.

## Why This Follows v1.x

The v1.x Delta Receiver line is already closed and frozen in
`docs/releases/v1.9.1.md` and
`docs/testing-v1.x-delta-receiver-closure-matrix.md`.

That line made receiver acceptance, Delta Handoff, usage packs, lint fixtures,
repair references, dogfooding, and adoption discoverability reviewable. It is
complete enough to serve as the input baseline for v2.0. Continuing to add
v1.9.x navigation patches would keep BaseBrief in closure work instead of
moving it toward the next useful artifact.

## Context Pack Lite Output

The planned artifact set is:

```text
MANIFEST.md
REPO_MAP.md
KEY_FILES.md
RECENT_DELTA.md
RISK_BOUNDARIES.md
RECEIVER_STATE.md
NEXT_WINDOW_STARTER.md
```

The pack should be local-first, rule-based, file-based, reviewable, and safe to
trim. It should prefer explicit known inputs over broad inference.

## Status Semantics

Context Pack Lite should preserve these semantics:

- `reviewed`: accepted or human-reviewed context
- `needs-review`: generated or inferred context awaiting review
- `generated`: mechanically produced from local files or git facts
- `not_available`: expected input was not found
- `not_applicable`: field does not apply to the repo or stage
- `stale`: inherited context that was not reverified in the current run

Sections should also name `source`, `trust`, and `stale` when that metadata
changes how a receiver should act.

## Degradation Rules

Context Pack Lite must not pretend the repo has complete input.

If Project State, Delta Handoff, receiver output, git range data, or a known
release doc is missing, the correct behavior is to mark the section as
`not_available`, `not_applicable`, or `needs-review`. Empty or minimal
`RECEIVER_STATE.md` is valid when the repository has no receiver history.

## Why Not Workflow Runner First

Workflow Runner Lite should follow Context Pack Lite, not lead it. A runner is
useful only after the core artifact is stable enough to generate and review.

Recommended order:

```text
v2.0 Context Pack Lite
v2.1 Context Pack Check
v2.2 Workflow Runner Lite
```

## Non-Goals

- No `context-pack` command in v2.0-A.
- No provider request.
- No AI automatic summary.
- No vector database, embedding, or semantic index.
- No runtime, daemon, or task queue.
- No plugin, MCP server, IDE integration, hosted service, or cloud-memory work.
- No `schema-v2`, `basebrief-project-state-v2`, or `basebrief-sidecar-v2`.
- No new checker rule family in this planning phase.
- No command output format change.
- No repo dump tool.
- No push, tag, release, pull request, npm publish, or global CLI install.

## Validation Gate

The local validation gate for this planning baseline is:

```text
node scripts/run_release_checks.js
npm run check
git diff --check
```

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```

## Next Implementation Slice

After review, the smallest implementation slice is a rule-based Context Pack
Lite generator under the existing `scripts/` structure. It should generate the
seven planned files, record live branch / HEAD / worktree facts, list key
files, include recent git facts when available, preserve risk boundaries, and
write a next-window starter. It should avoid a large `src/` migration until the
POC proves useful.

Minimal command shape:

```text
node scripts/basebrief.js context-pack --repo <target-repo> --output-dir <dir> [--since <commit>] [--max-files <n>] [--json]
```

The v2.0-B generator remains local-first and file-based. It does not introduce
schema-v2, provider requests, AI automatic summaries, vector indexes, runtime
behavior, plugin surfaces, MCP, IDE integration, or Workflow Runner behavior.
