# v3.0.0 Workflow Runner Lite Plan

Date: 2026-06-10

Status: v3.0-A research/spec

This document opens the v3.0 line after `v2.9.1` Project Profile / Recipes
Lite dogfooding. v3.0 should explore Workflow Runner Lite as a narrow local
BaseBrief pipeline runner, not as an automation platform.

This plan is research/spec only. It does not add a runner script, CLI command,
JSON contract, schema-v2, provider path, runtime integration, plugin, MCP
server/tools, daemon, watcher, CI workflow, or automatic project-task execution.

## Goal

Workflow Runner Lite should reduce manual command stitching across the existing
local BaseBrief surfaces:

```text
Project Profile -> Continue -> Context Pack -> Check -> Resume starter
```

The future command shape can be:

```text
node scripts/basebrief.js workflow --profile <profile.json> --output-dir <dir> [--repo <target-repo>] [--since <commit>] [--max-files <n>] [--json]
```

The command should prepare reviewable handoff artifacts and a clear next action.
It must not execute project work.

## Proposed Contract

The first future workflow contract should be:

```text
workflowContractVersion: basebrief-workflow-lite-v1
command: workflow
status: ready_for_receiver | needs_review | blocked
profile
recipe
profileDefaultsApplied
steps.continue
steps.check
steps.resume
outputFiles.report
outputFiles.contextPack
outputFiles.starter
```

Human output may name the next review action. JSON output must stay public-safe
and must not include the full resume prompt body or human-only `next_step`
fields.

## Candidate Step Semantics

- `continue`: create the continuation package from a profile or explicit repo
  options.
- `check`: validate the generated context pack through the existing check
  surface.
- `resume`: prepare a receiver starter from the checked pack.

The initial runner should not call Doctor, Export, Delta, Project State, git, or
release commands automatically. Those may remain explicit separate commands
until dogfooding proves a narrower need.

## Boundaries

- No provider request.
- No raw private output.
- No runtime integration.
- No plugin.
- No MCP server.
- No MCP tools.
- No schema-v2.
- No hosted service.
- No daemon.
- No watcher.
- No CI workflow.
- No global config.
- No secret store.
- No Project State schema change.
- No Context Pack seven-file structure change.
- No Doctor expansion.
- No Export expansion.
- No automatic project task implementation.
- No automatic commit, push, tag, release, pull request, npm publish, or global
  CLI install.

## Acceptance Matrix

The future implementation should prove:

```text
profile outside repo + workflow --profile -> ready_for_receiver or needs_review
dirty repo -> workflow -> needs_review, not automatic task execution
broken generated pack -> workflow -> blocked with check summary
workflow --json -> no prompt or next_step leak
explicit CLI flags -> override profile defaults
plain continue/context-pack/check/resume commands -> unchanged
```

## Validation Gate For This Plan

This planning step should verify only repository consistency:

```text
node --check scripts/run_release_checks.js
npm run release-check
git diff --check
```

Expected release-check output without provider env remains:

```text
provider_probe_status=skipped
```

## Rollback

If the v3.0 line starts becoming a hosted platform, background automation,
provider/runtime integration, schema-v2 migration, plugin/MCP surface, CI
workflow, or automatic project-task runner, stop and narrow it back to a local
reviewable BaseBrief pipeline spec before implementing code.
