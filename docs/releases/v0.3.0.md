# BaseBrief v0.3.0 Release Candidate

Status: local release candidate

Date: 2026-06-05

BaseBrief v0.3.0 turns receiver handoff from a readable convention into a bounded local workflow. It keeps the existing Chinese-first, Skill-first, Full/Lite public entry while adding explicit receiver-ready semantics and optional read-only repository verification.

## Highlights

### Receiver-ready v1

- Handoffs now separate `receiver_entry_task` from `post_acceptance_next_action`.
- `expected_changed_files` provides an exact, stable changed-file manifest.
- Source-window verification is distinguished from receiver-window re-verification.
- Receiver reports use explicit task, repository, declared-check, and acceptance statuses.
- Agent-authored response language follows the natural-language body of the latest user message, with `preferred_language` as fallback.

### Receiver Safe Check v1

- Independent `basebrief-receiver-check-v1` and `basebrief-receiver-check-result-v1` contracts.
- Optional `node_syntax`, `artifact_check`, and `file_tokens` checks.
- Exact branch, HEAD, and changed-file comparison.
- `pass` and `difference_found` exit zero; only `blocked` exits nonzero.
- Repository-relative path enforcement, real-path boundary checks, and `.env` / `.git` protection.

### Explicit Receiver Workflow

```text
node scripts/basebrief.js receiver-init --repo <target-repo> --output <receiver-check.json> --json
node scripts/basebrief.js receiver-check --config <receiver-check.json> --repo <target-repo> --json
```

`receiver-init` creates a state-only config for review. It does not overwrite files, write tracked target-repository files, or guess behavioral checks. `receiver-check` then performs the declared read-only verification.

## Compatibility

- BB9 handoff schema is unchanged.
- Builder, Adapter, Seal/Diff, provider profiles, and benchmark behavior are unchanged.
- Receiver Safe Check remains optional and disabled when `receiver_check_config` is `not_applicable`.
- Existing Full, Lite, and cache-ready routing remains unchanged.
- CLI Lite remains repository-local and zero dependency; v0.3.0 does not introduce an installed CLI, plugin, MCP server, hosted service, or account system.

## Evidence

Repository-reproducible validation for this release candidate includes:

- unit coverage for receiver init/check pass, difference, blocked, exit-code, path, symlink, ignored-file, detached-HEAD, and no-tracked-write behavior
- release checks covering the public schemas, CLI help, Quickstart workflow, examples, links, and publication safety
- Artifact Checker validation for new public receiver documentation and examples
- `git diff --check` and protected-interface review

No Codex receiver thread or full model matrix is required for this release candidate. Any external-agent smoke result is supplementary evidence only and is not cross-model stability proof.

One bounded OpenCode read-only documentation smoke completed and identified a source-window versus receiver-window role ambiguity in the Quickstart; the wording was corrected. One Claude Code smoke timed out and was not retried. These outcomes are execution records, not model-quality conclusions.

## Known Limits

- State matching does not prove runtime or behavioral correctness.
- Declared checks are intentionally narrow and are not a general test runner.
- `expected_changed_files` does not prove file-content integrity.
- `generated_at` does not automatically expire a handoff.
- BaseBrief does not automatically switch the receiver working directory.
- CRLF policy, content hashes, raw-command execution, prompt merging, plugins, and installed CLI packaging remain out of scope.

## Upgrade Notes

Existing v0.2.2 users can continue using readable Full/Lite handoffs without Receiver Safe Check.

To adopt receiver-ready v1:

1. Use the updated Full, Lite, and Next Chat templates.
2. Set `receiver_check_config: not_applicable` for state-only manual verification.
3. Use `receiver-init` when an explicit state snapshot is useful.
4. Review the generated config before adding declared checks.
5. Run `receiver-check` from the receiver workflow and report all four receiver statuses.

## Release Gate

The local release candidate is ready only after:

- all automated tests and release checks pass
- public artifacts report zero Artifact Checker errors and warnings
- protected interfaces remain unchanged
- commits contain no secrets or private absolute paths
- the user separately approves push, tag, and release

This document is not a release announcement. No push, tag, or formal release is performed by the release-candidate preparation task.
