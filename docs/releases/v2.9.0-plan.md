# v2.9.0 Project Profile / Recipes Lite Plan

Date: 2026-06-10

Status: v2.9-A project profile planning

This document opens the v2.9 line after `v2.8.0` Continuation Harness Lite.
v2.9 adds a small local Project Profile / Recipes layer so repeated
continuation runs can reuse reviewed defaults without turning BaseBrief into a
Workflow Runner or configuration platform.

## Goal

Project Profile / Recipes Lite stores public-safe local defaults for
continuation packages:

```text
profile-init -> review profile -> continue --profile
```

The profile should reduce repeated CLI argument typing and project-specific
recipe drift while keeping the generated continuation package reviewable.

## Command Shape

The v2.9 command shape is:

```text
node scripts/basebrief.js profile-init --repo <target-repo> --output <profile.json> [--recipe continuation-default|small-delta|review-heavy] [--json]
node scripts/basebrief.js continue --profile <profile.json> --output-dir <dir> [--repo <target-repo>] [--since <commit>] [--max-files <n>] [--json]
```

Explicit `continue` CLI flags override profile defaults.

## Profile Shape

The first profile contract is:

```text
schemaVersion: basebrief-project-profile-v1
recipe: continuation-default | small-delta | review-heavy
repo_hint: relative local repo hint
defaults.since
defaults.max_files
starter_language
risk_boundaries
review_notes
non_goals
```

The initial recipes only tune local defaults:

- `continuation-default`: normal next-window continuation packages
- `small-delta`: narrower recent-delta packages
- `review-heavy`: wider public-safe file review packages

## Safety Rules

Profiles must be public-safe and reviewable. They must reject:

- `.env` and `.git` paths
- private absolute paths
- secret-like values
- token, secret, credential, password, API key, OAuth, bearer, or env fields
- unsupported recipes
- invalid `max_files`

## Boundaries

- No provider request.
- No raw private output.
- No runtime integration.
- No plugin.
- No MCP server.
- No MCP tools.
- No schema-v2.
- No Workflow Runner.
- No global config.
- No secret store.
- No Project State schema change.
- No Context Pack seven-file structure change.
- No Doctor expansion.
- No Export expansion.
- No automatic project task implementation.
- No automatic commit, push, tag, release, pull request, npm publish, or global
  CLI install.

## Validation Matrix

The v2.9 implementation must verify:

```text
profile-init -> public-safe basebrief-project-profile-v1 JSON
continue --profile -> continuation package ready
CLI flags -> override profile defaults
unsafe profile field -> rejected
unsafe profile output path -> rejected
continue --repo -> unchanged from v2.8
continue --json with profile -> no prompt or next_step leak
```

The local validation gate is:

```text
node --check scripts/basebrief.js
node --check scripts/basebrief_project_profile.js
node --test tests/project-profile.test.js
npm test
npm run release-check
git diff --check
```

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```

## Rollback

If profile support starts storing secrets, global config, runtime details,
provider credentials, schema-v2 data, or automatic task/git actions, stop and
remove the profile command, profile script, tests, docs, example kit, and
release-check assertions. Keep `continue --repo` unchanged.
