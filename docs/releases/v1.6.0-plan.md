# v1.6.0 Delta Receiver Lint Fixture Pack Plan

Date: 2026-06-07

This document records the recommended v1.6.0 planning direction after the
v1.5 Delta Receiver Lint Mini line became locally closed.

It is a planning baseline, not a push, tag, release, package publication,
provider integration, runtime integration, plugin, MCP, IDE, hosted service,
Auto Flow behavior, global CLI install, npm publication, command output format
change, JSON schema claim, or schema-v2 claim.

## Summary

BaseBrief v1.6.0 should keep the receiver lint line local-first and
review-based by adding a copyable public fixture pack:

```text
Delta Receiver Lint Fixture Pack
```

The goal is not to add rules, commands, schemas, or runtime enforcement. The
goal is to make the v1.5 receiver lint rules easier to learn, copy, and explain
without requiring authors to infer rule families from tests.

## Recommended Direction

Prefer a docs/examples/tests slice:

- add `examples/receiver/lint/README.md` as the public fixture guide
- add public-safe clean, error, and warning fixtures under
  `examples/receiver/lint/`
- keep every failing fixture intentionally small and tied to one primary rule
  family
- document the expected checker status, severity, and rule ID for each fixture
- verify fixture behavior through existing `basebrief check`, release checks,
  and independent tests

## Fixture Coverage

The fixture pack should cover:

- clean pass
- delta-style missing report section
- starter-style missing `pass/fail`
- starter-style missing `wait for user confirmation`
- starter-style missing fact layer
- receiver result JSON consistency error
- `difference_found` semantics warning
- historical `commits_in_range` drift warning

## v1.6.0 In Scope

- Public-safe receiver lint fixture guide and examples.
- Documentation updates that point users from receiver usage and checker docs
  to the fixture pack.
- Release-check and test coverage that verifies expected pass, error, and
  warning behavior.

## v1.6.0 Out Of Scope

- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No `basebrief-project-state-v2`.
- No `basebrief-sidecar-v2`.
- No new CLI command.
- No new npm script.
- No machine-readable JSON schema.
- No command output format change.
- No Auto Flow.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output in public docs.

## Success Criteria

v1.6.0 is successful if all of the following are true:

1. A public user can run `basebrief check` against each fixture and see the
   expected rule family.
2. Clean fixtures pass with zero errors and zero warnings.
3. Error fixtures fail with the documented receiver rule family.
4. Warning fixtures pass while reporting the documented warning.
5. Fixture docs preserve the v1.5 semantics: `difference_found` remains a
   completed verification result, and historical `commits_in_range` drift
   remains non-blocking when refreshed branch, HEAD, and worktree facts match.
6. No schema, command, runtime, provider, plugin, MCP, IDE, Auto Flow, hosted,
   or publication scope is introduced.

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```
