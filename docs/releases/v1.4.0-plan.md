# v1.4.0 Delta Receiver Usage Pack Plan

Date: 2026-06-07

This document records the recommended v1.4.0 planning direction after the
v1.3 Delta Receiver Starter Integration line became locally closeout-ready.

It is a planning baseline, not a push, tag, release, package publication,
provider integration, runtime integration, plugin, MCP, IDE, hosted service,
Auto Flow behavior, global CLI install, npm publication, command output format
change, JSON schema claim, or schema-v2 claim.

## Summary

BaseBrief v1.4.0 should keep the Delta Receiver line local-first and
review-based by collecting the v1.1-v1.3 receiver discipline into one public
usage guide, one example router, and one copyable starter outline:

```text
Delta Receiver Usage Pack
```

The goal is not to create a new runtime, command, or schema. The goal is to
let a receiver see what to read, what to copy, and when to report `pass`,
`difference_found`, or `blocked` without flipping across several docs first.

## Recommended Direction

Prefer a docs/examples/tests slice:

- add one public usage guide with a short decision matrix and minimum read
  order
- add one example router that points to existing public-safe `pass`,
  `difference_found`, `blocked`, language-routing, and golden-path examples
- add one copyable starter outline that keeps machine fields and human anchors
  together
- keep `difference_found` as a completed verification result, not an agent
  failure
- keep historical `commits_in_range` drift non-blocking when refreshed branch,
  HEAD, and worktree facts still match
- keep source-window inherited facts, live repo facts, and receiver-window
  rechecks separate
- explain when to use the Delta report kit versus the golden-path starter
  report

## v1.4.0 In Scope

- Public receiver-facing documentation polish for the Delta Receiver line.
- A centralized example router and a copyable starter outline.
- Documentation discoverability, release-check coverage, and focused tests.

## v1.4.0 Out Of Scope

- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No `basebrief-project-state-v2`.
- No `basebrief-sidecar-v2`.
- No new CLI command.
- No new npm script.
- No machine-readable JSON schema.
- No command output format change.
- No Auto Flow.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output in public docs.

## Success Criteria

v1.4.0 is successful if all of the following are true:

1. A receiver can find one public usage entry before reading deeper dogfooding
   docs.
2. The usage pack tells the receiver when to use the Delta report kit and when
   to use the golden-path starter report.
3. Existing report semantics stay stable: `pass`, `difference_found`, and
   `blocked` keep their meanings; human `fail` can coexist with machine
   `difference_found`.
4. No schema, command, runtime, provider, plugin, MCP, IDE, Auto Flow, hosted,
   or publication scope is introduced.
5. `provider_probe_status=skipped` remains the no-provider validation gate when
   provider environment variables are absent.

## Deferred Alternatives

Machine-readable receiver report schemas, new receiver commands, runtime
integration, external receiver automation, and provider-backed tests remain
deferred until repeated local usage shows a concrete blocker.

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```
