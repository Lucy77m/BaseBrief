# v1.0.0 Planning Baseline (Adopted)

Date: 2026-06-07

This document records the adopted v1.0 direction after `v0.9.3` closed and
froze the `v0.9.x` line.

It is a planning baseline, not a release claim and not a request to start
`v1.1+` scope.

## Summary

BaseBrief should not keep stretching `v0.9.x`, and it should not jump straight
into a large plugin, runtime, or cloud direction.

The adopted direction is:

```text
BaseBrief v1.0 = Delta Handoff Compiler for coding agents
```

Internally, the quality bar remains:

```text
Reviewable Delta Handoff Compiler
```

That means the v1.0 line should focus on one core job: help a new AI window
understand what changed since the last accepted continuity point, with explicit
review status and stable local boundaries.

## Adopted Judgment

This plan combines the strongest parts of the current roadmap drafts:

- Keep the original product positioning:
  local-first, runtime-agnostic handoff compiler for coding agents.
- Keep the `Delta Handoff` line as the v1.0 core.
- Keep `reviewability` as a real standard, but do not let it expand v1.0 into a
  large provenance platform.
- Reuse the current `seal` / `diff` line instead of inventing a second,
  competing state-diff model.
- Keep the current state schema stable for v1.0 and store delta baseline data
  outside `basebrief-project-state-v1`.
- Keep v1.0 narrow, heavily dogfooded, and validated by real next-window use.

## Product Position

BaseBrief at v1.0 is still not:

- a repo packer
- a hosted memory platform
- a provider gateway
- an IDE plugin suite
- an agent runtime
- a task manager

It remains:

```text
project continuity artifacts
+ local project state
+ handoff delta
+ sidecar bundle
+ receiver contract
+ public-safe boundaries
```

## v1.0 In Scope

The adopted v1.0 scope is:

1. A `delta` command that turns:
   - current local state
   - git commit range facts
   - file change facts
   - state-level diffs from `seal` / `diff`
   into one reviewable `delta-handoff.md`.
2. A local `.basebrief/delta-baseline.json` file that tracks the next delta
   baseline without changing `basebrief-project-state-v1`.
3. Stable `reviewed` / `needs-review` semantics in the delta output so the next
   receiver can distinguish confirmed continuity from generated summaries.
4. Checker, tests, and release checks that cover the delta line.
5. Early self-dogfooding on the BaseBrief repo before calling v1.0 ready.
6. A release-candidate pass that proves a new AI window can continue from the
   delta handoff with much less baseline questioning.

## v1.0 Out Of Scope

The adopted v1.0 plan does not include:

- provider requests
- runtime integration
- schema breaking changes
- `basebrief-project-state-v2`
- `basebrief-sidecar-v2`
- repo-map auto scanning as a release blocker
- unified orchestration commands such as `run --mode ...` as a release blocker
- MCP server work
- IDE extensions
- cloud memory or hosted sync
- new benchmark lines
- plugin or platform work

## Key Design Decisions

### 1. Delta extends the current state-diff line

`delta` should build on top of `seal` / `diff`, not replace them with a second
parallel state-diff model.

Adopted direction:

- `seal` / `diff` remain the state-level change engine.
- `delta` aggregates:
  - state-level changes
  - git log range facts
  - changed-file facts
  - receiver-oriented narrative structure

This keeps the existing logic useful and avoids maintaining two competing diff
systems.

### 2. Keep `basebrief-project-state-v1` unchanged

Do not add delta-baseline fields directly to `.basebrief/state.json` in v1.0.

Adopted direction:

```text
.basebrief/state.json
.basebrief/history/
.basebrief/delta-baseline.json
```

`delta-baseline.json` carries delta-specific continuity, while `state.json`
continues to represent reviewed project state.

### 3. `.basebrief/out/` stays local and ignored

Generated delta and continuation artifacts should stay local-first and ignored
by default.

Adopted direction:

- `.basebrief/out/` is local output, not public history by default.
- public-safe examples belong under `examples/` or `docs/`, not under raw local
  output directories.

### 4. Dogfooding happens before the finish line

Do not wait until the final release-candidate step to test whether the delta
artifact actually helps a new AI window.

Adopted direction:

- once the minimum `delta` command works on BaseBrief itself, run a real
  next-window continuation exercise immediately
- fix template or data-shape issues there
- only then continue toward v1.0 release hardening

### 5. Full provenance remains optional for the first v1.0 slice

The v1.0 line should keep reviewability as a standard, but it should not get
blocked by a heavyweight provenance subsystem debate before delta itself proves
useful.

Adopted direction:

- the main release bar is clear reviewable delta output
- richer provenance structure may exist later in v1.0 only if it helps the
  delta line directly
- it is not allowed to displace the core delta objective

## Recommended Phases

### Phase 0: Direction Freeze

Write and freeze the planning/spec baseline:

- `docs/releases/v1.0.0-plan.md`
- `docs/specs/delta-handoff.md`
- any small supporting spec notes needed to settle delta-baseline storage and
  review semantics

Exit condition:

- v1.0 scope is explicit
- no major design ambiguity remains around delta vs state lifecycle

### Phase 1: Delta Spec And Example

Define the target artifact before coding:

- `delta-handoff.md` structure
- section sources
- review semantics
- command shape
- one public-safe example

Exit condition:

- the spec is concrete enough to test against

### Phase 2: Minimal Delta CLI

Implement the narrowest useful path:

- read current local state
- read delta baseline
- gather git range facts
- reuse `seal` / `diff`
- write `delta-handoff.md`
- update `delta-baseline.json`
- run checker/tests

Exit condition:

- BaseBrief can generate a valid delta handoff locally

### Phase 3: Self-Dogfooding And Repair

Run the delta flow on this repository and feed the result to a fresh AI window.

Exit condition:

- the receiver no longer needs to ask basic baseline questions such as
  "what project is this?" or "what is the current goal?" before useful work

### Phase 4: v1.0 RC Hardening

After the delta line is already useful:

- tighten docs
- tighten validation
- tighten release checks
- verify the continuation line again end to end

Exit condition:

- the v1.0 line is locally release-ready

## Success Criteria

v1.0 is successful if all of the following are true:

1. `delta` works on the BaseBrief repository without provider access.
2. The output clearly separates reviewed continuity from generated change
   summary content.
3. `delta-baseline.json` advances predictably between runs.
4. The checker and tests guard the new delta line.
5. A fresh AI window can read the delta continuation package and quickly restate:
   - current goal
   - recent key changes
   - at least one active risk boundary
   - next implementation slice
6. v1.0 does not widen into runtime, provider, plugin, or schema work.

## First Implementation Slice

The first concrete implementation slice after this planning baseline should be:

1. write `docs/specs/delta-handoff.md`
2. add one public-safe example delta artifact
3. implement the minimal `delta` command
4. add `delta-baseline.json`
5. add focused tests
6. dogfood the result on BaseBrief itself

## Current Boundaries

The current continuation boundaries remain:

- no push / tag / release / publish required for v1.0 planning work
- no provider request
- no runtime integration
- no schema breaking change
- no raw private output exposure
- no cloud memory expansion
- no automatic widening into `v1.1+`
