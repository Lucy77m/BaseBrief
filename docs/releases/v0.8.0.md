# v0.8.0 Sidecar Handoff Bundle Candidate

This candidate adds a local sidecar handoff bundle on top of the existing
Project State lifecycle.

Command shape:

```text
node scripts/basebrief.js sidecar-build --repo <target-repo> [--target generic|openclaw] [--output-dir <dir>] [--json]
```

## What Changed

- Adds `sidecar-build` to read an existing valid `.basebrief/state.json`.
- Writes `handoff.md`, `next-chat-prompt.md`, `receiver-entry-task.md`,
  `risk-boundaries.md`, `state-summary.json`, and `manifest.json`.
- Keeps `generic` as the default target.
- Adds an `openclaw` target with stronger safety wording only.
- Refuses non-empty output directories; there is no force overwrite flag.

## Boundaries

- No provider request.
- No raw private output.
- No runtime integration.
- No schema change.
- No Auto Flow.
- No receiver thread creation.
- No OpenClaw/Hermes runtime connection.
- No OpenClaw/Hermes profile, config, memory, or workspace writes.
- No push, tag, formal release, or npm publish in this local candidate.
- Wait for user confirmation before continuing from the generated bundle.

## Project State Contract

`sidecar-build` is a consumer of `basebrief-project-state-v1`. It does not
change `.basebrief/state.json`, does not call `state-advance`, and does not add
new Project State fields.

Default output is `.basebrief/sidecar/<target>/`. Dogfooding and tests should
use ignored private output directories and keep public records to public-safe
summaries only.

## Validation Plan

- `node --test tests/basebrief.test.js`
- `node scripts/run_release_checks.js`
- `npm run check`
- `git diff --check`
- Artifact check against ignored private sidecar output.

Provider validation is intentionally skipped for this candidate:
`provider_probe_status=skipped`.
