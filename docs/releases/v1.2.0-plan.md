# v1.2.0 Delta Receiver Report Kit Plan

Date: 2026-06-07

This document records the adopted v1.2.0 planning baseline after the v1.1
Delta Receiver Acceptance line was locally closed.

It is a planning baseline, not a push, tag, release, package publication,
provider integration, runtime integration, plugin, MCP, IDE, hosted service,
Auto Flow behavior, global CLI install, npm publication, or schema-v2 claim.

## Summary

BaseBrief v1.2.0 should turn the v1.1 receiver acceptance discipline into a
small public-safe report kit:

```text
Delta Receiver Report Kit
```

The goal is to help a receiver window produce the same minimum acceptance
report every time. The kit is a Markdown/text reporting aid for humans and
agents; it is not a JSON schema, command output contract, or automated runtime.

## Standard Report Fields

A Delta Receiver report should include these fields:

- `receiver_task_status`
- `repository_state_status`
- `handoff_acceptance`
- `blocking_or_repair_notes`
- `current_goal`
- `live_repo_state`
- `inherited_fact_differences`
- `hard_boundaries`
- `next_narrow_slice`

The report should separate source-window inherited facts from receiver-window
rechecks before it recommends the next narrow slice.

## Acceptance Semantics

- `handoff_acceptance: pass` means the entry verification completed and live
  branch, HEAD, and worktree facts matched the refreshed handoff facts.
- `handoff_acceptance: difference_found` means the entry verification completed
  and accurately reported a live-state difference. It is not an agent failure.
- `handoff_acceptance: blocked` means the receiver could not complete necessary
  verification safely.

Historical dry-run or pre-commit `commits_in_range` values may remain as facts
from the time they were recorded. A later receiver should treat explainable
historical count drift as non-blocking when refreshed `delta-handoff.md` branch,
HEAD, and worktree facts match live repository state.

## v1.2.0 In Scope

- A public-safe planning baseline for the report kit.
- A public-safe dogfooding record for the report kit.
- Two public-safe example reports: one `pass`, one `difference_found`.
- Documentation index and testing docs discoverability.
- Release-check and focused test coverage for fields, semantics, examples, and
  boundaries.

## v1.2.0 Out Of Scope

- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No `basebrief-project-state-v2`.
- No `basebrief-sidecar-v2`.
- No new CLI command.
- No new npm script.
- No machine-readable JSON schema for receiver reports.
- No command output format change.
- No Auto Flow.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output in public docs.

## Success Criteria

v1.2.0 is successful if all of the following are true:

1. A receiver can copy the report shape without inventing fields.
2. The examples demonstrate both `pass` and `difference_found`.
3. The examples preserve source-window facts versus receiver-window rechecks.
4. The examples explain non-blocking historical count drift without rewriting
   historical evidence.
5. `basebrief-project-state-v1`, `basebrief-delta-handoff-v1`, and
   `basebrief-delta-baseline-v1` remain unchanged.

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```
