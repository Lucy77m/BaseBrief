# v1.8.0 Delta Receiver Lint Dogfooding Evidence Plan

Date: 2026-06-07

This document records the recommended v1.8.0 planning direction after the
v1.7 Delta Receiver Lint Repair Pack line became locally closed.

It is a planning baseline, not a push, tag, release, package publication,
provider integration, runtime integration, plugin, MCP, IDE, hosted service,
Auto Flow behavior, global CLI install, npm publication, command output format
change, JSON schema claim, schema-v2 claim, or checker rule change.

## Summary

BaseBrief v1.8.0 should keep the receiver lint line local-first and
review-based by recording public-safe dogfooding evidence:

```text
Delta Receiver Lint Dogfooding Evidence
```

The goal is not to broaden automation. The goal is to verify that the v1.6
fixtures and v1.7 repair guide work against existing public receiver examples.

## Recommended Direction

Prefer an evidence/docs/tests slice:

- add `docs/dogfooding/delta-receiver-lint-dogfooding-v1.8.md`
- record only public-safe command shapes, checker status, and friction notes
- include v1.6 intentional fixture outcomes, v1.7 fixed example outcomes, and
  existing receiver example outcomes
- keep raw private output, provider calls, external runners, and new automation
  out of scope
- verify the dogfooding record with Artifact Checker, release checks, and
  independent tests

## v1.8.0 In Scope

- Public-safe dogfooding evidence for receiver lint fixtures and repair
  examples.
- Documentation updates that route users from usage docs to fixtures, repairs,
  and evidence.
- Release-check and test coverage for evidence discoverability and safety.

## v1.8.0 Out Of Scope

- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No `basebrief-project-state-v2`.
- No `basebrief-sidecar-v2`.
- No new CLI command.
- No new npm script.
- No machine-readable JSON schema.
- No command output format change.
- No Auto Flow.
- No checker rule change.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output in public docs.

## Success Criteria

v1.8.0 is successful if all of the following are true:

1. The dogfooding record verifies v1.6 fixture behavior and v1.7 repair
   behavior with public-safe evidence.
2. Existing receiver examples still pass Artifact Checker.
3. The evidence records friction without raw private output.
4. No schema, command, runtime, provider, plugin, MCP, IDE, Auto Flow, hosted,
   checker-rule, or publication scope is introduced.

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```
