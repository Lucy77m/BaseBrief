# v0.8.7 Copyable New-Window Starter Candidate

`v0.8.7` is a Sidecar usability patch. It adds a first-class
`new-window-starter.md` artifact so users have one clean block to copy into a
new chat. It does not add a new CLI command, API, schema, runtime integration,
Auto Flow, provider smoke, tag, release, or npm publish step.

## Changes

- `sidecar-build` now writes `new-window-starter.md` for both `generic` and
  `openclaw` targets.
- `manifest.json` declares `output_files.newWindowStarter` for newly generated
  bundles.
- Human CLI output reports `new_window_starter=<path>`.
- `sidecar-check` validates the starter when the manifest declares it, while
  old v0.8 bundles without that manifest key remain compatible.
- Public docs now tell future BaseBrief final responses to include:

```text
新窗口开场白（可复制）：
<contents of new-window-starter.md>
```

## Required Starter Content

`new-window-starter.md` must include a target repository cue, a Sidecar bundle
instruction, `current_goal`, `receiver_entry_task`, at least two risk
boundaries, report pass/fail, Wait for user confirmation, No provider request,
No raw private output, No runtime integration, No schema change, and
No auto-advance.

For `openclaw`, the starter must also preserve OpenClaw/Hermes runtime and
profile/config/memory/workspace write prohibitions.

## Boundaries

- No provider request.
- No raw private output.
- No runtime integration.
- No schema change.
- No Auto Flow.
- Wait for user confirmation.
- No clipboard automation.
- No UI integration.

`sidecar-build` remains a local Project State consumer for
`basebrief-project-state-v1`. `sidecar-check` remains a read-only structure and
public-safety gate for `basebrief-sidecar-v1`.

## Release Gate

The no-provider release gate remains:

```text
provider_probe_status=skipped
```
