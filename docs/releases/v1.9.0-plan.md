# v1.9.0 Delta Receiver Lint Discoverability / Adoption Plan

Date: 2026-06-07

This document plans a narrow adoption polish pass for the locally closed v1.6
fixture pack, v1.7 repair pack, and v1.8 dogfooding evidence line.

It is not a push, tag, release, package publication, provider integration,
runtime integration, plugin, MCP, IDE, hosted service, platform claim, Auto
Flow behavior, global CLI install, npm publication, command output format
change, JSON schema claim, schema-v2 claim, checker rule change, or new rule
family.

## Goal

Make the existing receiver lint public surface easier to find, read in the
right order, and map back to the existing receiver examples.

This pass should reduce three public author mistakes:

- copying an intentionally broken fixture as if it were a valid handoff
- stopping at a rule ID instead of opening the fixed repair reference
- missing the existing pass, `difference_found`, blocked, language-routing, and
  starter-facing receiver examples after lint repair

## Adoption Path

The recommended public read order is:

1. Start with `docs/receiver-usage-pack.md` to choose the receiver route.
2. Open `examples/receiver/usage-pack/README.md` to pick the closest existing
   public receiver example.
3. Use `examples/receiver/lint/README.md` only when a receiver lint rule or
   intentionally broken shape needs explanation.
4. Use `examples/receiver/lint/repair/README.md` to copy the smallest fixed
   shape after a finding.
5. Return to the existing receiver examples for the final report shape:
   `delta-report-pass`, `delta-report-difference-found`, `blocked`,
   `language-routing`, or golden-path starter reports.
6. Read `docs/dogfooding/delta-receiver-lint-dogfooding-v1.8.md` only when
   public-safe evidence for the fixture and repair line is needed.

## Mapping Table

| If you have | First compare | Then copy from |
| --- | --- | --- |
| no finding, Delta acceptance passed | `clean-pass-receiver-report.md` | `examples/receiver/delta-report-pass/README.md` |
| live mismatch after completed verification | `difference-found-warning-receiver-report.md` plus `fixed-delta-receiver-report.md` | `examples/receiver/delta-report-difference-found/README.md` |
| unsafe or incomplete verification | usage-pack route notes | `examples/receiver/blocked/README.md` |
| starter-facing report missing anchors or fact layers | starter failing fixtures plus `fixed-starter-report.md` | `examples/golden-path/first-pass-receiver-report.md` or `follow-up-receiver-report.md` |
| receiver result JSON consistency issue | `json-invalid-result-consistency.json` plus `fixed-result.json` | `examples/receiver/difference-found/receiver-check-result.json` when the completed outcome is `difference_found` |
| language routing question | usage-pack route notes | `examples/receiver/language-routing/README.md` |

## Implementation Scope

Allowed changes:

- add this plan document
- add navigation from `docs/index.md`, `docs/testing.md`,
  `docs/receiver-usage-pack.md`, `docs/checks.md`, and the roadmap
- make `examples/receiver/usage-pack/README.md` route readers through the
  fixture -> repair -> existing example sequence
- make `examples/receiver/lint/README.md` and
  `examples/receiver/lint/repair/README.md` warn clearly which files are
  learning fixtures versus copyable fixed references
- add release-check and independent-test assertions that the adoption path stays
  visible

Out of scope:

- No checker rule change.
- No new rule family.
- No new CLI command.
- No new npm script.
- No machine-readable JSON schema.
- No command output format change.
- No provider request.
- No runtime integration.
- No plugin, MCP, IDE, hosted, or cloud-memory work.
- No schema-v2 work.
- No push, tag, release, pull request, npm publish, or global CLI install.
- No raw private output copied into public docs.

## Validation Plan

The validation gate for this adoption polish pass is:

```text
node scripts/run_release_checks.js
npm run check
git diff --check
```

When provider environment variables are absent, release checks must keep:

```text
provider_probe_status=skipped
```

## Exit Criteria

- a public reader can start at the usage pack and find the fixture, repair, and
  existing receiver example sequence without guessing
- intentionally broken fixtures are clearly marked as learning inputs, not
  copyable receiver reports
- fixed repair references point back to existing public receiver examples
- `difference_found` remains a completed verification result, not an agent
  failure
- historical `commits_in_range` drift remains non-blocking when refreshed
  branch, HEAD, and worktree facts still match live repository state
